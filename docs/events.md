# SpringX JSONL Event Protocol

SpringX writes line-delimited JSON events to stdout so the WebUI can follow a scan without parsing human log text. Human logs remain enabled by default for compatibility with the current WebUI.

Use `--log-format jsonl` or `--jsonl-only` when stdout must contain only JSONL events. In that mode, Banner, disclaimer, and human-readable log lines are suppressed; human logs are emitted as `log` events instead.

## Envelope

Every event uses the same envelope:

```json
{
  "version": "springx.events.v1",
  "type": "service_detected",
  "scan_id": "187b8d6c2d8c0000",
  "seq": 3,
  "time": "2026-06-25T10:15:30.1234567+08:00",
  "data": {},
  "error": ""
}
```

- `version`: protocol version. Current value is `springx.events.v1`.
- `type`: event name.
- `scan_id`: stable ID for one `scan` command execution.
- `seq`: monotonically increasing sequence number within one process.
- `time`: RFC3339 timestamp generated by Go JSON encoding.
- `data`: event-specific payload.
- `error`: optional error string for failed events.

## Event Types

### `scan_started`

Emitted after arguments are parsed and the scan ID is created.

Data:

```json
{
  "id": "187b8d6c2d8c0000",
  "args": ["scan", "-u", "https://example.com", "--nopoc"]
}
```

### `target_discovered`

Emitted for each normalized URL target before probing.

Data:

```json
{
  "url": "https://example.com"
}
```

### `log`

Emitted instead of human-readable stdout log lines when `--log-format jsonl` or `--jsonl-only` is enabled.

Data:

```json
{
  "message": "[INF] SpringX self-owned scanner core v0.1.0-mvp"
}
```

### `service_detected`

Emitted after an HTTP/HTTPS service is identified.

Data includes:

```json
{
  "host": "example.com",
  "ip": "93.184.216.34",
  "port": 443,
  "protocol": "WEB应用",
  "scheme": "https",
  "url": "https://example.com",
  "status_code": 200,
  "title": "Example Domain",
  "server": "ECS",
  "tls": "TLS1.3",
  "content_type": "text/html",
  "content_length": 1256,
  "location": "",
  "favicon_hash": "123456789",
  "technologies": ["Nginx"],
  "fingerprint_sources": ["server:nginx"]
}
```

### `port_open`

Emitted for an open TCP port. Payload follows the same shape as `service_detected`.

### `poc_started`

Emitted before Nuclei starts.

Data:

```json
{
  "engine": "nuclei",
  "targets": 1,
  "template_dir": "D:\\github_projects\\SpringX-Scanner\\pocs\\nuclei",
  "tags": ["cve"],
  "severity": "critical,high",
  "ids": ["springx-smoke"]
}
```

### `vulnerability_found`

Emitted for each Nuclei finding. The `engine`, `template_id`, `severity`,
`target`, and `name` fields are always present; the remaining fields are
included only when non-empty so the live stream stays compact. The WebUI uses
these to render a detail expansion (matched path, description, request/response
summaries, extracted values) in real time, without waiting for the final report
JSON.

Data:

```json
{
  "engine": "nuclei",
  "template_id": "springx-smoke",
  "severity": "info",
  "target": "http://127.0.0.1:8080",
  "name": "SpringX local smoke finding",
  "matched_at": "http://127.0.0.1:8080/springx-smoke",
  "type": "http",
  "description": "...",
  "matcher_name": "...",
  "extractor_name": "...",
  "extracted_results": ["..."],
  "request_summary": "GET /springx-smoke ...",
  "response_summary": "HTTP/1.1 200 ...",
  "timestamp": "2026-06-30T10:11:12+08:00"
}
```

Only `engine`, `template_id`, `severity`, `target`, and `name` are guaranteed;
the other fields are omitted when empty.

### `poc_completed`

Emitted after POC execution, including skipped cases.

Data:

```json
{
  "engine": "nuclei",
  "targets": 1,
  "findings": 1,
  "duration": "2.351s",
  "template_dir": "D:\\github_projects\\SpringX-Scanner\\testdata\\nuclei",
  "skipped": false,
  "skip_reason": "",
  "error": ""
}
```

### `scan_completed`

Emitted when the scan runner reaches a terminal non-failed state.

Data:

```json
{
  "id": "187b8d6c2d8c0000",
  "status": "completed"
}
```

`status` may be `completed` or `stopped`.

### `scan_failed`

Emitted when target parsing, port parsing, or another fatal scan error stops execution.

Data:

```json
{
  "id": "187b8d6c2d8c0000"
}
```

The `error` field contains the failure message.

### `report_written`

Emitted by the CLI wrapper after HTML, Markdown, and JSON reports are written.

Data:

```json
{
  "html": "D:\\github_projects\\SpringX-Scanner\\reports\\html\\SpringX-Scan-20260625-101530.html",
  "markdown": "D:\\github_projects\\SpringX-Scanner\\reports\\markdown\\SpringX-Scan-20260625-101530.md",
  "json": "D:\\github_projects\\SpringX-Scanner\\reports\\data\\SpringX-Scan-20260625-101530.json"
}
```

`report_written` can appear after `scan_completed` because report rendering happens after the scan runner returns. Consumers should wait for process exit or this event before refreshing the report list.
